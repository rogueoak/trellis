# Language

How to write anything public-facing for rogueoak: READMEs, docs, release notes, landing copy,
and the strings a user actually reads. Internal notes and code comments can be looser. This
file is about the voice the world sees.

## Voice

Warm and human, specific, with a light touch of dry wit. Write to a reader who is sharp but new
to *this* thing, not new to software. Give enough context to follow along, then stop. Do not
labour a point once it has landed.

## Rules

1. **Address the reader as "you." Skip the first person.** No "we", "I", or "our" in product
   copy.
   - Yes: "Run `/trellis-install` and you are set."
   - No:  "We built trellis-install so we could set you up."
2. **Be terse.** Cut any word that is not pulling weight. Shorter almost always reads better.
   - Yes: "Trellis ships the rules. Adopt them in one command."
   - No:  "Trellis is a framework designed to ship a curated collection of rules that..."
3. **Show, with examples.** A concrete example beats a paragraph about the example. When you
   explain a thing, give one.
4. **Dry wit is welcome, sparingly.** One good aside per page, not one per line. If you are not
   sure it lands, cut it. A clean sentence never hurts.
5. **No marketing hype.** Drop "revolutionary", "game-changing", "seamless", "blazing-fast",
   "effortless", "powerful". Say what it does and let that carry the weight.
   - Yes: "Installs in one command, updates in another."
   - No:  "A revolutionary, seamless way to supercharge your repos."
6. **No buzzwords or empty jargon.** If a term adds no meaning, remove it. Spell out an acronym
   the first time it appears.
7. **Give enough context, no more.** The reader is smart but does not live in your head. One
   sentence of setup is usually plenty; three is usually too many.
8. **ASCII only.** No em-dashes or en-dashes (see `guidelines.md`). Plain quotes and hyphens.

## Quick test

Before you publish, read it once and ask: would this annoy a sharp friend who is short on time?
If yes, cut until it would not.
